### 2026-06-10 18:38 — Code Voice: concurrent turns crashed the TTS server (the REAL cause of "mixed results")
The dominant reason voice notes were inconsistently missing: mlx Kokoro is NOT thread-safe, and `server.py` uses `ThreadingHTTPServer` with synthesis done directly in each request thread. When two+ sessions end turns close together (overseer + forecast in parallel), their `/v1/audio/speech` calls hit `model.generate()` concurrently and **crash the whole server process**; launchd restarts it ~1s later but every in-flight request dies with `RemoteDisconnected` → no note for any of them. A turn ending in isolation delivered fine — hence "mixed." Detaching synthesis (the 17:50 fix) made it worse by letting children overlap. Reproduced cleanly: 3 concurrent children → all RemoteDisconnected + a `Code Voice server starting` line in server.log at the same second (crash+restart). Fixed with `_synth_lock` serializing all synthesis (3 concurrent → ok/ok/ok, no restart) + client retry/backoff in `say_to_phone.synth_mp3` + 120→180s client timeout. Debug tip: missing notes → check `server.log` for `RemoteDisconnected`/unexpected `server starting` lines (crash) vs `Broken pipe` (the older 15s-timeout bug). Ceiling: synthesis is now serial (~13-35s each), so ~5+ sessions ending simultaneously could still exceed the 180s client timeout for the last in line; if that bites, move to a proper bounded queue with longer waits.

### 2026-06-10 17:50 — Code Voice: long voice notes silently dropped (15s daemon-thread cliff)
The real reason long summaries "never became voice messages": `stop_hook.py` ran synthesis in a daemon thread with `t.join(timeout=15)`. Long notes (2-3k chars) take 20-40s to synth on the Kokoro server; at 15s the hook process exited, killed the daemon thread mid-request, and the socket broke (`server.log`: `[Errno 32] Broken pipe`) — no mp3, no send. Short notes (<15s) delivered fine, so it masqueraded as clipping. The earlier "speak the whole message" fix made it WORSE by pushing text past the 15s cliff. Fixed by spawning `say_to_phone.py` as a fully detached subprocess (`start_new_session=True`, text via stdin): hook returns in 0.45s, child synthesizes independently (verified 2376-char note → 37.6s → telegram ok=True, completes after hook exit, no broken pipe). Debug tip preserved in README: long notes missing → grep `server.log` for "Broken pipe" first. Open follow-up: many sessions firing detached helpers concurrently all hit the one shared (non-thread-safe) Kokoro model on :8765 — not observed failing, but a serialization/lock there may be needed if voice notes start corrupting under load.
