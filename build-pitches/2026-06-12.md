## PITCH 1 (Primary): Loop Engineering — Design Goals, Not Prompts

**Technique:** Instead of manually prompting Claude Code to do tasks, you design *loops* — an autonomous agent cycle with a trigger and a verifiable goal. The agent starts itself, works, and continues until the goal is provably met (tests pass, QC score above threshold, spec fully implemented). Your job shifts from "prompt the agent" to "specify what done looks like."

**Who's doing it:**
- **Boris Cherny** (creator of Claude Code, Anthropic) in a widely-circulated interview: *"I don't prompt Claude anymore. I have loops that are running. They're the ones that are prompting Claude and kind of figuring out what to do. My job is to write loops."*
  - Source: https://www.productmarketfit.tech/p/stop-prompting-ai-and-start-building
- **Peter Steinberger** (creator of OpenClaw), tweet that hit 5 million views in under 24 hours (June 7, 2026): *"Here's your monthly reminder that you shouldn't be prompting coding agents anymore. You should be designing loops that prompt your agents."*
- **Addy Osmani** (Google) published a synthesis framework on loop engineering within days: https://addyosmani.com/blog/loop-engineering/
- **YouTube coverage** ("Only the best are using them…", June 9, 2026): https://www.youtube.com/watch?v=dMrm2jAyrKM

**Evidence it's real:**
- Claude Code ships `/loop` as a documented, live command: `/loop every 5 minutes "compare what we have with spec.md and keep building until complete."` — real, in-product today.
- The SWE-Marathon benchmark (Rishi Desai, June 6, 2026) quantified loop-based agent performance at 26% on hard novel tasks — first hard metric on autonomous loop effectiveness.
- Multiple practitioner guides published independently within a week, indicating real adoption, not just hype. Boris Cherny reports running "hundreds of Claude instances simultaneously" with autonomous loops monitoring GitHub issues and Slack.
- This isn't a new technique in theory (it's reinforcement-learning-adjacent) but the combination of Claude Code's `/loop` command + verifiable goals is now specifically productized and within reach.

**Why it matters for us:**
Kyle's current pipeline (`generate-episode.sh`) calls `claude -p` with one-shot prompts and wraps each step in a hard timeout. That's the *old* meta. The loop paradigm maps directly onto this stack:

- **QC loop:** Instead of "run QC agent, read result, fix manually, re-run QC," define the goal as "a QC agent passes clean" and let Claude Code loop until it does. Right now QC failure means human intervention. A loop makes it auto-healing.
- **Episode generation loop:** "Generate a complete Killen Time script that passes QC and covers all ingest items." The agent runs, drafts, QC-checks, revises — autonomously — until the criterion is met. Kyle wakes up to a finished, verified script, not a timed-out partial.
- **Observer-system loops:** Loops triggered by filesystem events (new transcript in .tmp/, new build-pitch file written) that process the item without a human kicking off the next step.
- **COS delegation:** The COS currently dispatches one-shot workers. Upgrading those workers to loop-based agents means delegating a *goal* not a *task* — the worker doesn't come back to report until it's done.

**Build sketch (an agent could execute this cold):**
1. Add a `QC_LOOP` wrapper to `generate-episode.sh`: replace the current one-shot QC step with `claude --loop "Run QC on scripts/killen-time-$(date +%Y-%m-%d).txt, fix all issues found, and repeat until QC passes with zero issues found."` with a 20-minute ceiling.
2. Document the goal specification pattern in `CLAUDE.md` under "Loop-based pipeline steps" — so every future pipeline step that currently says "run X and manually fix" can be upgraded.
3. For the COS dispatcher: add a `--loop` mode flag to the `delegate` skill that wraps the worker briefing in a loop goal rather than a one-shot prompt. The COS says "build X until the PR is green" not "write some code for X."
4. Experiment: `/loop every 5 minutes "Check .tmp/ for new unprocessed transcripts. If found, pick the highest-scored one, write a segment draft to .tmp/segments/, and mark it processed."` — passive autonomous draft generation during ingest.

## PITCH 2: Cloudflare's Specialized-Agent + Coordinator Code Review Architecture

**Technique:** Replace a single generalist code-review agent with N specialized sub-agents (each with a focused system prompt for one domain: security, performance, correctness, documentation, architecture) plus a coordinator/orchestrator agent that collects their outputs, deduplicates findings, and posts one structured review. Sub-agents write findings as JSONL for real-time streaming observability. Trigger is CI/PR open; coordinator's single custom tool `spawn_reviewers` kicks off the whole pipeline.

**Who's doing it:**
- **Cloudflare**, documented by engineer Ryan Skidmore: https://blog.cloudflare.com/ai-code-review/
- **IndyDevDan YouTube breakdown** ("I Ranked Cloudflare's Software Factory," June 8, 2026): https://www.youtube.com/watch?v=YG4t7aMY81c
- Built on **OpenCode** (open-source agent framework with a programmatic SDK — a Claude Code alternative with CI-native access)

**Evidence it's real:**
Production metrics from Cloudflare's own 30-day audit (March 10–April 9, 2026):
- **131,246 review runs** across **48,095 merge requests** and **5,000+ codebases**
- **Median review time: 3 minutes 39 seconds** (vs. hours for human review)
- **Average cost per review: $1.19** (median $0.98) — achieved through 85.7% prompt cache hit rate
- **159,103 findings** (~1.2 per review), with hard-block mode for critical security issues
- **55% increase in merge request velocity** over 4-week rolling average (5,600/week → 8,700/week)
- 93% adoption across Cloudflare engineering teams

The architecture is explicitly published and replicable; it's not a black-box internal tool.

**Key implementation details from the transcript:**
- **7 specialized agents**: security, performance, code quality, documentation, release management, compliance, plus internal engineering codecs
- **Coordinator agent** with a single custom tool (`spawn_reviewers`) that kicks off the pipeline — coordinator deduplicates and synthesizes into one comment
- **Risk-tiered dispatch**: trivial diffs get 2 agents; security-sensitive files trigger all 7
- **JSONL output** from sub-agents for streaming observability — never wait for a full JSON blob, process events as they stream
- **Plugin-based config**: per-codebase customization via a composable plugin architecture loaded from JSON config

**Why it matters for us:**
Kyle has a `code-reviewer` skill (single agent) that reviews PRs and posts findings. The Cloudflare pattern would upgrade it to specialized sub-agents per concern with far better signal-to-noise. More concretely:
- A **security reviewer sub-agent** would catch injection, credential exposure, SSRF — things a generalist reviewer may downweight
- A **correctness reviewer** focusing only on "does this logic work" gives cleaner findings than one agent trying to do everything
- The **JSONL streaming** pattern is directly applicable to any agent in the stack that produces long outputs — the observer compiler, the COS, the episode QC agent. Start acting on findings before the full output lands.
- The plugin config pattern is applicable to `beats.json` — per-beat agent configuration with a composable lifecycle.

**Build sketch (an agent could execute this cold):**
1. Refactor the `code-reviewer` skill: instead of one `general-purpose` agent, spawn 3 focused sub-agents in parallel — (a) security-and-injection reviewer, (b) logic-and-correctness reviewer, (c) architecture-and-test-coverage reviewer. Each gets a focused system prompt, each returns JSONL findings.
2. Add a coordinator step that reads all three JSONL outputs, deduplicates by file+line, weights by severity, and composes one `gh pr review` comment with structured sections.
3. Set a `risk_tier` based on files changed: if `*.py` auth/config/credentials paths touched, trigger all 3 sub-agents; otherwise use just logic+architecture for trivial changes.
4. Estimated cost: at $1-2 per review (Cloudflare's real number), this is effectively free for Kyle's volume. Token spend dominated by prompt caching — make sure sub-agents share a cached system context prefix.

**Status: pitched — 2026-06-12**
