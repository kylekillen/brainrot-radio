This is sane UX design, but the blockage surfaces once you start letting the agent iterate nightly on legacy web monorepos assembled over seven colleagues in four time zones—you inevitably drift into patching commits you didn’t author when auto-fixing regressions. Second angle: the emergent git-for-agents debate that surfaced on Hacker News under the title “git will eat the agent economy.” Two competing standalone repositories appeared in daylight hours yesterday alone. First is xjdr’s git-for-agents—the forked repo adds schematized metadata entries to every commit, so agentic toolchains record intent-state convergence deltas in addition to the usual diff hunks. The metadata schema embeds: original prompt, agent-engine step index, confidence score, external-tool decisions list, and rollback checksum. Those five fields are compressed via zstd and appended to commit footer. Achievable within existing git plumbing, no extra files.

[BASIL]
The second repo is cuspflow’s agit-backend—a completely new porcelain using sqlite backend so agents can dereference history by intent hash instead of commit SHA. The interaction model looks like “git-intent-refactor verify 0.8 confidence” and returns a pointer to the historic commit with lambda-prob > 0.8 the agent believes matches. Mongo-style document versioning layered over git. Downsides: breaks every existing hook, pushes custom porcelain to every contributor, and violates git clone remote expectations. Upside: semantic search across intent trees is immediate—could compare every attempt where the “transpile nested routes to rust” task was attempted across repo lifetimes.

[BROOKE]
Smart take from the Hacker thread with 231 thumbs up: either git eats agents or agents eat git. There is no middle ground. If git stays unchanged, agent developers will migrate to specialized VCS layers because precision roll-forward becomes impossible on arbitrary commit histories. If git adaptation wins—either through xjdr or cuspflow—the canonical version-control protocol embedded across 100 million repositories bends around agentic workflows rather than human keyboards. That is foundational infrastructure change happening at module-level granularity in realtime.

We move to measurement. A new benchmark domain dropped after midnight at aacl-lin and the full YAML files are already mirrored on GitHub. The tag: AA-Briefcase-benchmark-v0.2. Pre-release report is forty-seven pages, but the core idea is a single design axiom I think many of us missed when we rolled our own eval harnesses. Here is the axiom: “Long-horizon agent tasks only matter to the extent that you can reproduce the exact agent context state-mask across iterations. Therefore all agent evals are simultaneously benchmark and checkpointing utility, not competitive racing.” In plain English, if your eval doesn’t snapshot the KV-cache plus active thread state instantly restartable at powers of two minutes, you’re measuring something else—probably prompt coherence—not agent coherence.

[BROOKE]
The benchmark exposes thirty-two realistic, long-tail coding problems whose average solution horizon is between three hours and seventy-six minutes. Each project downloads a target repo at a stable hash, sets up Docker container family, then tasks any agent to push from the stable branch to a feature branch that decisively solves the repo’s open issue without introducing regression. Once the build completes and tests green, the agent halts. The critical twist: the harness then freezes the agent Spider-Man log (σημασιολγικό repr: adds root hash + tempfile pointers) into a checkpoint record. You can now rerun the exact same trajectories on later agent harnesses with no hallucination-risk artifact because the checkpoint encapsulates not the command but the conditioned foundation model state. Zero-noise OOD benchmark.

[BASIL]
Preliminary numbers posted: Claude-Fable-5 achieved 63 % pass rate once, reset to checkpoint on version 5.2, rate improved to 81 %. The delta is almost entirely attributable to restored context recall—agent didn’t need to rediscover evolved patch motivations. AA-Briefcase argues you therefore cannot publish frontier-agent evals as absolute scores unless you also publish at-scale checkpoint replay instructions. The repo’s lead, Jason Hashway, writes in the README: “Agent reproducibility without state-level serialization is like publishing Whetstone MIPS scores without CPU microarchitecture.” The directive to personal setups: if you plan to measure agents in multi-task day-to-day usage, your task harness better write out transient-state checkpoints or every iteration is effectively spinning random race conditions.
